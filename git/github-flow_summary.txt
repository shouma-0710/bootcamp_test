■GitHub flow の理解

1. GitHub flow の概要と特徴
  - メインとなる main ブランチと作業用ブランチの2種類だけで運用するシンプルなワークフロー。
  - 構造が明瞭で分かりやすく、小規模から大規模まで迅速に開発を進めたいときに適している。

2. git-flow との違い
  - GitHub flow は2種類のブランチのみを使用する。
  - 一方、git-flow は以下の5種類のブランチを使い分けるため、複雑な大規模開発に適している。
    - main (旧 master) ブランチ：リリース済みのソースコードを管理する。
    - develop ブランチ：次のリリースに備えた最新のソースコードを管理する。
    - feature ブランチ：新機能開発やバグ修正を行う（develop から分岐、develop へマージ）。
    - release ブランチ：リリース作業を行う（develop から分岐、main と develop へマージ）。
    - hotfix ブランチ：緊急の修正作業を行う（main から分岐、main と develop へマージ）。

3. GitHub flow における6つの原則（ルール）
  - main ブランチのものは何であれ常にデプロイ可能である：
    マージ後すぐにデプロイすることを前提に、テストやレビューを慎重に行う必要がある。
  - main から説明的なブランチを作成する：
    開発時は必ず main から分岐し、何の作業をしているか一目で分かる名前の作業ブランチを作る。
  - 名前をつけたブランチに定期的に push する：
    ローカルの変更を定期的にリモートへ push し、メンバー間の衝突トラブルを未然に防ぐ。
  - いつでも Pull Request を作成する：
    マージ時だけでなく、開発途中でフィードバックやアドバイスが欲しいときにも PR を作成する。
  - マージは Pull Request がレビューされた後だけ行う：
    main ブランチを常にデプロイ可能に保つため、他者によるレビュー通過を必須とする。
  - レビューの後は直ちにデプロイする：
    放置による他メンバーとのマージ衝突を防ぎ、ワークフローの信頼性を高く保つ。

4. 開発の具体的な流れ
  - 【1】作業用ブランチの作成と開発：
    分かりやすい名前で作業ブランチを作成し、定期的にリモートリポジトリへ push しながら進める。
  - 【2】Pull Request (PR) の実行：
    マージの準備が整った段階、または開発途中でアドバイスが欲しい段階で PR を作成する。
  - 【3】コードレビューの実施：
    PR を受け取ったメンバーがレビューを行い、問題がなければ main ブランチへマージする。
  - 【4】直ちに行うデプロイ：
    main へのマージ完了後、他の変更との衝突を避けるために速やかにデプロイを行う。
